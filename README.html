A production-grade Learning Management System — JWT-authenticated REST API backed by SQL Server stored procedures, fronted by an Angular 21 SPA with role-based access control.

C# / .NET 10 Angular 21 (TypeScript) SQL Server JWT + Refresh Tokens Role-Based Access
01 — Overview
Architecture Overview

BrightPath is a two-tier application: an ASP.NET Core Web API backend and an Angular 21 SPA frontend. Both live under LearningApi/. The backend uses a strict four-layer architecture (Controller → Service → Repository → Database). All SQL access goes through stored procedures — no inline SQL in C#. The frontend uses Angular's functional-style interceptors, route guards, and Angular Signals for reactive state.

Backend (ASP.NET Core)
  • Target framework: net10.0
  • Auth: JWT Bearer + Refresh Tokens (SHA-256 hashed, stored in DB)
  • Password hashing: BCrypt.Net-Next
  • Data access: Raw ADO.NET via SqlCommand + SqlDataReader
  • All SQL: 100% stored procedures, zero inline SQL
  • Rate limiting: ASP.NET Core built-in, 5 req/min on auth endpoints
  • Error handling: GlobalExceptionMiddleware catches all unhandled exceptions
  • Response envelope: All responses wrapped in ApiResponse<T>
  • Docs: Swagger / OpenAPI with JWT Bearer security scheme
Frontend (Angular 21)
  • Framework: Angular 21 standalone components
  • State: Angular Signals + toSignal() from RxJS interop
  • HTTP: HttpClient with functional interceptors
  • Auth flow: JWT stored in localStorage, expiry decoded client-side
  • Token refresh: Silent refresh with request queuing via BehaviorSubject
  • Routing guards: authGuard (login check) + roleGuard (Admin-only pages)
  • Icons: lucide-angular v1
  • Testing: Vitest (unit tests in *.spec.ts)
  • Build: Angular CLI 21 / esbuild

06 — Backend
DTOs & Validation

All input validation lives on DTOs via Data Annotations. Program.cs overrides ApiBehaviorOptions.InvalidModelStateResponseFactory so ModelState failures return the standard ApiResponse envelope instead of default ASP.NET validation problem format.

DTOKey Constraints
RegisterDto Username: [StringLength(100, min=2)]; Email: [EmailAddress]; Password: [StringLength(100, min=8)] + regex requiring uppercase, lowercase, digit
LoginDto Email: [Required][EmailAddress]; Password: [Required]
CourseCreateDto / CourseUpdateDto CourseName: [StringLength(200, min=2)] + regex (must contain letters, allows digits/spaces/special chars); Fee: [Range(1, 1000000)]; DurationWeeks: [Range(1, 52)]
StudentCreateDto / StudentUpdateDto Name: [StringLength(100)] + regex (letters, spaces, hyphens, apostrophes, dots); Email: [EmailAddress]
EnrollmentCreateDto StudentID + CourseID: both [Required][Range(1, int.MaxValue)]
PagedResultDto<T> Output-only wrapper: Items, TotalCount, Page, PageSize
ApiResponse<T> Universal envelope: Success (bool), Message (string), Data (T?)
C# ResponseHelper.cs — wraps every response
public static ApiResponse<T?> Success<T>(T? data, string message = "") => new { Success = true, Message = message, Data = data }; public static ApiResponse<T?> Fail<T>(string message) => new { Success = false, Message = message, Data = default }; // Overloads exist for data+message Fail (used by ModelState errors)
07 — Backend
Service Layer

Services contain business logic and throw custom exceptions that the GlobalExceptionMiddleware converts to HTTP responses. Each service is registered via AddScoped against its interface.

AuthService
  • Register: BCrypt.HashPassword → calls repo
  • Login: CleanupExpiredTokens → GetByEmail → BCrypt.Verify → GenerateToken (HMAC-SHA256, 2hr expiry) → GenerateRefreshToken (64 random bytes → base64) → SHA-256 hash → RevokeExisting → Save new
  • Refresh: ValidateRefreshToken → GetUserById → GenerateToken → RotateRefreshToken (revoke old, save new hash)
  • Logout: RevokeRefreshToken by hash
  • AssignRoleSafely: Guards against removing last admin
CourseService / StudentService
  • GetAll: Returns list of ResponseDtos
  • GetPaged: Delegates to repo with page/pageSize/search
  • GetById: Throws NotFoundException if null
  • Add: Returns new ID from repo
  • Update: Throws NotFoundException if 0 rows affected
  • Delete: Validates ID > 0, throws BadRequestException / NotFoundException
EnrollmentService
  • GetAll: Returns joined list (StudentName, CourseName, Date)
  • GetPaged: Search across student name + course name
  • Add: Forwards to repo — duplicate prevention is in SQL proc
  • Delete: Same pattern as other services
08 — Backend
Repository Layer

Every repository reads the connection string from IConfiguration. All calls use SqlCommand with CommandType.StoredProcedure and typed SqlParameter — no string interpolation, no inline SQL.

Duplicate key handling: Repos catch SqlException where ex.Number == 2601 || ex.Number == 2627 (unique constraint violations) and re-throw as BadRequestException. This covers email/CourseName uniqueness.
RepositoryInterface Methods
AuthRepository Register, GetByEmail, GetById, UpdateEmail, UpdateRole, SaveRefreshToken, GetRefreshToken, RevokeRefreshToken, CleanupExpiredRefreshTokens, RevokeActiveRefreshTokensForUser, GetAllUsers, CountAdmins
CourseRepository GetAll, GetPaged, GetById, Add (returns new ID via SCOPE_IDENTITY()), Update, Delete
StudentRepository GetAll, GetPaged, GetById, Add, Update, Delete
EnrollmentRepository GetAll, GetPaged, Add, Delete
AdminAuditLogRepository Add, GetPaged
Paged queries pattern: All paged stored procedures return two result sets — first is the data rows, second is SELECT COUNT(*) AS TotalCount. The repository reads both using reader.NextResult() and packs them into PagedResultDto<T>.

10 — Backend
Auth System

Full JWT + Refresh Token implementation. Access tokens expire in 2 hours. Refresh tokens expire in 7 days, stored as SHA-256 hashes in dbo.RefreshTokens. On login, all previous refresh tokens for the user are revoked first (single active session per user policy).

POST /api/auth/register
BCrypt hash password → RegisterUsers proc → role forced to "User"
POST /api/auth/login
Cleanup expired tokens → fetch user → BCrypt.Verify → generate JWT (HMAC-SHA256) + refresh token (64 random bytes) → SHA-256 hash refresh token → revoke old → save new
POST /api/auth/refresh
Validate hash match + expiry + IsRevoked=0 → fetch user → generate new access + refresh tokens → rotate (revoke old, save new hash)
POST /api/auth/logout
SHA-256 hash incoming token → RevokeRefreshToken proc (sets IsRevoked=1)
PUT /api/auth/assign-role (Admin)
Validates role ∈ AppRoles.All (case-insensitive) → AssignRoleSafely guards last-admin removal → writes audit log
C# JWT Token Claims
var claims = new[] { new Claim(ClaimTypes.Name, user.Username), new Claim(ClaimTypes.Email, user.Email), new Claim("UserID", user.UserID.ToString()), new Claim(ClaimTypes.Role, user.Role) }; // Expires: DateTime.UtcNow.AddHours(2) // Algorithm: HmacSha256 // Issuer/Audience read from appsettings.json
C# AppRoles.cs
public static class AppRoles { public const string Admin = "Admin"; public const string User = "User"; public static readonly HashSet<string> All = new(StringComparer.OrdinalIgnoreCase) { Admin, User }; }
Rate limiting on auth endpoints: [EnableRateLimiting("AuthPolicy")] on register, login, and refresh — 5 requests/minute per remote IP. HTTP 429 on breach.
11 — Backend
Global Exception Middleware

GlobalExceptionMiddleware wraps the entire pipeline. It catches any unhandled exception, logs it with the HTTP method, path, and trace ID, then maps exception type to HTTP status code via a C# switch expression:

C# GlobalExceptionMiddleware.cs
int statusCode = ex switch { BadRequestException => 400, NotFoundException => 404, UnauthorizedException => 401, _ => 500 }; // 500 responses use a generic message ("An unexpected error occurred") // to avoid leaking internal details. // context.Response.HasStarted guard prevents replacing in-flight responses.

16 — Frontend
HTTP Interceptors
authInterceptor
  • Bypasses login/register/refresh endpoints entirely (no token attached)
  • Attaches Authorization: Bearer {token} to all other outbound requests
  • On 401: checks for x-retry header (prevents infinite loop) → fetches refresh token → calls /api/auth/refresh using HttpBackend (bypasses interceptor chain)
  • Queue logic: If a refresh is already in-flight, subsequent 401 requests queue on a BehaviorSubject and resume when the refresh completes
  • On refresh failure: auth.logout() → navigate to /login
loaderInterceptor
Calls LoaderService.show() before a request goes out and LoaderService.hide() in the finalize() operator — so the spinner always hides even on error. LoaderService uses a reference counter (signal<number>) so overlapping requests don't hide the spinner prematurely.

Frontend Services
ServiceKey Responsibilities
AuthService login (stores JWT + user + refreshToken in localStorage), register, logout (local clear), logoutFromServer (POST logout → revoke server token), isLoggedIn (decodes JWT exp claim), getRole, isAdmin, assignRole
BaseApiService Abstract base with protected get<T>, post<T>, putWithMessage, delete<T> — all unwrap the ApiResponse<T> envelope. Throws if res.success === false.

StudentService Same pattern as CourseService — BehaviorSubjects for students$ and totalCount$.
EnrollmentService Same pattern — enrollments$ and totalCount$.
ToastService Uses a signal<Toast | null>. show(msg, type, duration=3000) sets the signal then setTimeout clears it. Exposes success(), error(), info() shortcuts.
ConfirmService Uses a signal<ConfirmState | null>. ask(message) returns a Promise<boolean> that resolves when user clicks confirm or cancel. UI binds to state signal.
LoaderService Reference-counted spinner state. show() increments counter; hide() decrements — only sets loading signal to false when counter reaches 0.
AdminAuditLogService Fetches paginated audit logs via GET /api/admin-audit-logs.
18 — Frontend
Page Components
Login / Register
Both use Angular ReactiveFormsModule with FormBuilder.nonNullable.group(). Login uses ChangeDetectionStrategy.OnPush and manually calls cdr.markForCheck() after async operations. Server-side validation errors (data field of ApiResponse) are displayed per-field via getFieldError() utility.
Dashboard
Uses forkJoin to parallel-load students (paged) + all courses + all enrollments. State managed via toSignal() subscriptions to the service BehaviorSubjects. Includes server-side search and pagination for students. Delete flow goes through ConfirmService.ask().
Courses
Server-side search + pagination (page, pageSize via signals). setQuery() resets page to 1 and re-fetches. totalPages computed from totalCount$ and pageSize. Delete uses ConfirmService. Create/Edit navigates to CourseForm page.
Enrollments
Same search/pagination pattern as Courses. Displays student name + course name + enrollment date. Add navigates to EnrollmentForm. EnrollmentForm loads all students and all courses for dropdowns, submits StudentID + CourseID.
StudentForm / CourseForm
Dual-mode (create/edit): checks for :id route param. If present, fetches existing record and patches form. Submits POST or PUT accordingly. Client-side validation mirrors backend DTO constraints.
AdminAuditLogs / Users
Admin-only pages. AdminAuditLogs shows paged, searchable table of all admin actions with admin name, email, IP, entity, and timestamp. Users page shows all registered users; admin can call assignRole() to promote/demote.
